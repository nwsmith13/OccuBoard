# OccuBoard

OccuBoard is an AI-assisted job search workspace built by ARSO. It helps job seekers organize roles, tailor resumes, draft recruiter messages, and keep application momentum visible without feeling like a CRM.

## Stack

- React + Vite
- Tailwind CSS
- React Router
- Zustand workspace store
- Supabase Auth, Postgres, Row Level Security, and Storage
- Serverless OpenAI route at `api/generate.js`
- PDF/DOCX resume import and export utilities lazy-loaded on demand

## Local Setup

```bash
npm install
npm run dev
```

Create `.env` from `.env.example`:

```bash
VITE_SUPABASE_URL=your_supabase_project_url
VITE_SUPABASE_ANON_KEY=your_supabase_anon_key

# Serverless/API route only. Never prefix this with VITE_.
OPENAI_API_KEY=your_server_only_openai_api_key
OPENAI_MODEL=gpt-5-mini

# Stripe test mode billing.
VITE_STRIPE_PUBLISHABLE_KEY=pk_test_...
STRIPE_SECRET_KEY=sk_test_...
STRIPE_WEBHOOK_SECRET=whsec_...
STRIPE_PRO_PRICE_ID=price_1Te8BxDhoHP54GBjlCbY2coq
APP_URL=http://localhost:5173

# Required for server-side Stripe webhook writes to Supabase.
SUPABASE_SERVICE_ROLE_KEY=your_server_only_service_role_key
```

If `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` are missing, the app falls back to local demo mode. Auth forms will open the local workspace and data is stored in browser `localStorage`.

For local AI testing, Vite mounts a development-only `/api/generate` middleware in `vite.config.js`. The OpenAI key still stays in Node/server runtime, not browser code.

## Environment Variables

| Name | Scope | Required | Notes |
| --- | --- | --- | --- |
| `VITE_SUPABASE_URL` | Browser-safe | Production yes | Public Supabase project URL. Vite exposes this to browser code. |
| `VITE_SUPABASE_ANON_KEY` | Browser-safe | Production yes | Public Supabase anon key. Use RLS policies to protect user data. |
| `OPENAI_API_KEY` | Server only | AI yes | Must be set only in serverless/deployment env settings. Never prefix with `VITE_`. |
| `OPENAI_MODEL` | Server only | No | Defaults to the app's configured model if omitted. |
| `VITE_STRIPE_PUBLISHABLE_KEY` | Browser-safe | Billing yes | Stripe test publishable key. Present for future Stripe.js usage; Checkout currently redirects using a server-created URL. |
| `STRIPE_SECRET_KEY` | Server only | Billing yes | Stripe test secret key for Checkout and Billing Portal. Never prefix with `VITE_`. |
| `STRIPE_WEBHOOK_SECRET` | Server only | Billing yes | Stripe webhook signing secret for `/api/stripe-webhook`. |
| `STRIPE_PRO_PRICE_ID` | Server only | Billing yes | Defaults to `price_1Te8BxDhoHP54GBjlCbY2coq` if omitted. |
| `APP_URL` | Server only | Billing yes | Base URL used for Stripe success, cancel, and portal return URLs. |
| `SUPABASE_SERVICE_ROLE_KEY` | Server only | Billing webhooks yes | Required for webhook subscription updates. Never expose this to browser code. |

Do not create `VITE_OPENAI_API_KEY`, `VITE_OPENAI_KEY`, or any other `VITE_` OpenAI secret. Vite embeds `VITE_` variables into client bundles.

## Supabase Setup

Run `supabase/schema.sql` in the Supabase SQL editor for a fresh project. It creates the core tables, triggers, RLS policies, and the private `resumes` storage bucket.

For older databases, apply these migrations if they have not already been run:

```sql
-- supabase/ai_guidance_migration.sql
-- supabase/resume_uploads_migration.sql
-- supabase/resume_versions_updated_at_migration.sql
-- supabase/billing_subscriptions_usage_migration.sql
```

Production requirements:

- Enable Supabase Auth providers you plan to support.
- Add production and preview app URLs to Supabase Auth URL configuration.
- Set the Site URL to the deployed app URL.
- Add redirect URLs such as:
  - `https://your-domain.com`
  - `https://your-domain.com/login`
  - `https://your-domain.com/app/dashboard`
  - Vercel/Netlify preview URLs if previews are used
- Keep RLS enabled on all user-owned tables.
- Confirm the `resumes` storage bucket is private.
- Confirm storage policies restrict files to `auth.uid()` folder ownership.

The frontend uses only the public Supabase anon key. Data protection depends on RLS and storage policies, not secrecy of the anon key.

## OpenAI Setup

`api/generate.js` reads `OPENAI_API_KEY` from server-side environment variables and calls OpenAI from the serverless/API runtime. The frontend calls only `/api/generate`.

Production requirements:

- Set `OPENAI_API_KEY` in the hosting provider's server/runtime environment settings.
- Set `OPENAI_MODEL` only if overriding the default.
- Do not store OpenAI keys in committed files.
- Rotate any OpenAI key that was ever pasted into `.env`, logs, chat, screenshots, or shared build output.

If the key is missing, users see a friendly "AI generation is not configured yet" message.

## Deployment

### Vercel

This repo includes a Vercel-compatible serverless route at `api/generate.js`.

1. Import the repo into Vercel.
2. Set the environment variables listed above.
3. Use the default Vite build command:

```bash
npm run build
```

4. Use `dist` as the output directory.
5. Add the deployed Vercel URL to Supabase Auth redirect URLs.

### Netlify

The React/Vite frontend can deploy normally to Netlify, but `api/generate.js` is not a Netlify Function as-is.

Options:

- Move the same handler logic into a Netlify Function and keep the browser request path mapped to `/api/generate`.
- Or deploy the API route separately and proxy `/api/generate` to it.

Set the same environment variables in Netlify, and add Netlify production/preview URLs to Supabase Auth redirect URLs.

## Security Checklist

- `.env`, `.env.local`, `dist`, and `node_modules` are ignored by git.
- No OpenAI key should appear in frontend code, `dist`, README, or committed files.
- No `VITE_OPENAI_*` variables should exist.
- Supabase anon key is public by design; RLS must be enabled and tested.
- Resume text and job descriptions are sensitive user data. Do not log them.
- Keep the `resumes` storage bucket private.
- Rotate exposed or uncertain OpenAI keys before deployment.

## Production Behavior

- Missing Supabase env vars: app runs in local demo mode.
- Missing OpenAI key: AI generation returns a friendly configuration error.
- Failed auth: Supabase error is shown in the auth form.
- Failed upload: import card shows a readable upload/parsing error.
- Failed export: export panel shows a fallback message and suggests Print Preview.
- Missing resume/profile data: AI tools ask the user to complete the needed profile details.

## Follow-Up Reminder Testing

Follow-up reminders are in-app only for now; there are no system, browser, calendar, or email notifications.

1. Open a job detail modal and set the follow-up date to today.
2. Confirm Dashboard Today's Focus shows the role and Applications shows `Follow up today`.
3. Set the follow-up date to yesterday and confirm the overdue state appears.
4. Snooze the follow-up to tomorrow and confirm `Snoozed until ...` appears.
5. Mark the follow-up completed and confirm `Completed` appears.
6. Use `Generate follow-up message` in the modal follow-up area and confirm the message is from the candidate to the company/recruiter.

## Current MVP Scope

Implemented:

- Public landing page
- Supabase sign up, login, logout, session persistence wiring
- Protected app routes
- Responsive app layout with collapsible sidebar
- Dashboard, New Jobs, Generated Resumes, Applications, Resume Studio, Messages, and Settings pages
- Profile and base resume setup saved to `profiles`
- Job intake with soft company/title guardrails
- Drag-and-drop application pipeline with persisted stage updates
- AI fit analysis, tailored resume drafts, and recruiter messages through a serverless route
- AI generation histories saved to `job_scores`, `resume_versions`, and `messages`
- PDF/DOCX/TXT resume import
- PDF/DOCX/Print Preview resume export
- Initial Supabase schema and migrations

Not implemented yet:

- Auto-applying workflows
- Email or recruiter-message integrations
- Full resume template system
- Payments
